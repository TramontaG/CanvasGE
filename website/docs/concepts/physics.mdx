Sliver’s physics is a scene-owned rigid-body pipeline.

- Motion is integrated with a real time step (`dt`).
- Each `GameObject` contributes one rigid body, and each hitbox on that object becomes a shape on that body.
- The scene runs broad phase, narrow phase, contact generation, and contact solving after your objects tick.
- Gameplay callbacks stay on `GameObject`, so you work with the same high-level scene/object API throughout the engine.

For most games, you work with `Scene`, `GameObject`, hitboxes, `speed`, `angularVelocity`, and `setPhisics(...)`.

## High-level physics API on `GameObject`

Use `setPhisics(...)` to configure how a `GameObject` behaves in the physics world:

```ts
gameObject.setPhisics({
  immovable: false,
  affectedByGravity: true,
  restitution: 0.2,
  friction: 0.4,
  mass: 2,
});
```

Physics settings:

- `immovable: true` maps the object to a static body.
- `immovable: false` maps it to a dynamic body.
- `affectedByGravity` controls whether scene gravity is applied during integration.
- `restitution` controls bounce.
- `friction` is copied into both static and dynamic friction on the physics body.
- `mass` is used to compute the body’s mass and inertia from its hitboxes.

## Units and timing

`speed` and `angularVelocity` are integrated with `dt`.

- `speed` is a linear velocity in world units per second.
- `angularVelocity` is a rotational velocity per second in the same angle unit as `rotation`.
- `Scene.tick(dt?)` passes `dt` into the physics world.
- If you do not provide `dt`, the scene derives it from the game tick rate (`1 / tickRate`).

Example:

```ts
import { Vector } from "sliver-engine";

gameObject.speed = new Vector(120, 0); // 120 px/s
```

At `60` ticks per second, that moves about `2` pixels each tick because the physics world integrates `speed * dt`.

Scene gravity is also time-based acceleration:

```ts
scene.setGravity(new Vector(0, 10));
```

That means a dynamic body with `affectedByGravity: true` gains roughly `10 * dt` units/s of vertical velocity each step.

## Hitboxes, Compound Bodies, and Rotation

Sliver uses `SquareHitbox` and `CircleHitbox` on `GameObject`s:

```ts
import { GameObject, SquareHitbox, Vector } from "sliver-engine";

const crate = new GameObject("crate", new Vector(100, 100));
crate.setPhisics({ immovable: false, mass: 2 });
crate.addHitbox(
  new SquareHitbox(Vector.zero(), new Vector(32, 32), crate, {
    solid: true,
    debug: false,
  })
);
```

How hitboxes behave:

- One `GameObject` becomes one rigid body.
- Every hitbox on that object becomes a shape on that same body.
- Multiple hitboxes behave as a single compound body.
- Physics rotation uses the same pivot as rendering, so detached hitboxes rotate coherently with the visible object.

Hitbox options still matter:

- `solid: true` means the hitbox participates in physical collision response.
- `solid: false` means the hitbox is a sensor/trigger only.
- `debug: true` still draws the hitbox overlay.

## What Happens Each Scene Tick

Each `Scene.tick()` does this:

1. Calls `tick()` on every `GameObject`.
2. Builds or updates one rigid body per object in the scene physics world.
3. Copies the authored object state (`position`, `rotation`, `speed`, `angularVelocity`, `phisics`) into those bodies.
4. Integrates dynamic bodies with gravity and `dt`.
5. Runs broad phase candidate generation from body + shape bounds.
6. Runs narrow phase contact manifold generation.
7. Calls `beforeColision(...)` and `onColision(...)` on gameplay objects.
8. Solves solid contacts.
9. Writes solved body state back to each `GameObject`.

Your own `tick()` functions still run first, and the scene physics step produces the resulting motion and collision response afterward.

## Sensors, Callbacks, and Suppression

The gameplay collision hooks still work the same way conceptually:

- `beforeColision(other)` is the filter hook.
- `onColision(other, penetration)` is the overlap notification hook.

Collision rules:

- If either side returns `false` from `beforeColision`, the pair is suppressed.
- Suppressed pairs do not receive `onColision`.
- Suppressed pairs do not enter the solid solver.
- `solid: false` hitboxes still generate overlap callbacks, but they do not generate physical response.

Example trigger:

```ts
trigger.addHitbox(
  new SquareHitbox(Vector.zero(), new Vector(64, 64), trigger, {
    solid: false,
  })
);
```

Example filter:

```ts
override beforeColision(other): boolean {
  return other.name !== "player-bullet";
}
```

The `penetration` passed to `onColision` comes from the contact manifold normal and penetration depth for that overlap.

## Inspecting Physics Results

Scenes expose the last physics step result:

```ts
const result = scene.getLastPhysicsStepResult();

console.log(result.candidatePairs);
console.log(result.sensorManifolds);
console.log(result.solidManifolds);
console.log(result.notifiedPairIds);
console.log(result.suppressedPairIds);
```

This is useful for:

- debugging sensors vs solid contacts
- checking whether a pair was suppressed by `beforeColision`
- inspecting which pairs were actually solved
- writing deterministic physics tests

## Advanced APIs

Most games should stay with `Scene` + `GameObject` + hitboxes.

If you need more control than the scene/object API exposes, Sliver also exports lower-level physics pieces such as:

- explicit `PhysicsBody` creation
- `kinematic` body types
- collision category/mask filtering
- direct access to `ScenePhysicsWorld`
- custom gameplay collision stepping or solver configuration

Those APIs are exported directly from `"sliver-engine"`, for example:

```ts
import {
  PhysicsBody,
  ScenePhysicsWorld,
  stepGameplayCollisions,
} from "sliver-engine";
```
